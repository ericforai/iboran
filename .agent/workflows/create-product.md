# 产品详情页生成工作流 (v2.0 Enhanced)

从产品文档、需求清单或竞品链接，生成符合“交付型”标准的高质量产品详情页。
**核心定位**：回答“我们靠什么交付、怎么交付、交付到什么程度”，拒绝空泛，强调**深度**与**证据**。

---

## Step 1: 深度提取与定义

根据输入资料，提取以下 12 个核心变量。**如果资料缺失，必须进行推理或搜索补充，不可留空。**

| 变量 | 说明 | 示例/要求 |
|------|------|-----------|
| **{产品名}** | 核心产品词 | 如“用友 YonSuite” |
| **{一句话定位}** | 价值主张 | “基于 AI 的企业级费控平台，实现业财税资档一体化” |
| **{目标人群}** | 规模/角色 | “年营收 5-50亿的集团型企业财务总监” |
| **{痛点场景}** | 4-5个具体痛点 | 必须包含：症状描述 + 负面影响（如：对账慢导致月结延迟3天） |
| **{架构逻辑}** | 系统拓扑 | 输入/处理/输出的逻辑流程，需生成 Mermaid 或图片 |
| **{核心能力}** | 6-9项深度功能 | **必须配图**（界面或示意图），拒绝纯文字列表 |
| **{交付方法}** | 实施路线图 | 诊断 -> 蓝图 -> 上线 -> 运维 的标准化流程 |
| **{集成规范}** | 技术规格 | 支持的协议（API/EDI）、集成对象（ERP/OA/银行） |
| **{量化价值}** | 3-4条硬指标 | 必须带数字，如“效率提升 80%”，不可用“大幅提升” |
| **{精选案例}** | 2-3个完整故事 | 结构：客户背景 -> 面临挑战 -> 解决方案 -> 最终成效 |
| **{常见问题}** | 5-6条FAQ | 覆盖：合规性、部署周期、费用模式、售后服务 |
| **{资源下载}** | 白皮书/视频 | 提供深度阅读材料的入口 |

## Step 3: 12段式页面结构

产品页采用 **高密度、强信任** 的 12 段式结构：

| 序号 | 模块名称 | 对应组件文件 | 内容重点 |
|------|----------|--------------|----------|
| 0 | **SEO** | `page.tsx` | JSON-LD, Metadata |
| 1 | **沉浸式 Hero** | `Hero.tsx` | 高清背景图/视频 + 价值主张 + 双CTA |
| 2 | **场景痛点** | `PainPoints.tsx` | **不做简单列表**，使用场景化卡片（症状-后果） |
| 3 | **解决方案架构** | `Architecture.tsx` | **[NEW]** 架构图/拓扑图，展示系统全貌 |
| 4 | **深度功能** | `Capabilities.tsx` | Tab切换/左右布局，**必须配图** |
| 5 | **交付方法论** | `Methodology.tsx` | **[NEW]** 可视化实施路径，体现专业交付能力 |
| 6 | **交付范围** | `DeliveryScope.tsx` | 明确包含/可选/不包含 |
| 7 | **技术规格** | `TechSpecs.tsx` | 集成能力、部署环境、安全合规 |
| 8 | **客户案例** | `CustomerStories.tsx` | **[UPGRADE]** 深度案例卡片（Logo + 证言 + 数据） |
| 9 | **资源中心** | `Resources.tsx` | **[NEW]** 白皮书、演示视频下载 |
| 10 | **FAQ** | `FAQ.tsx` | **[NEW]** 消除客户疑虑 |
| 11 | **底部 CTA** | `CTASection.tsx` | 强引导表单 |

## Step 4: 组件开发规范 (High Quality)

### 4.1 视觉与交互
- **UI模拟规范 (New Standard)**：
    - **首选方案**：对于仪表盘、数据看板等界面，**强制使用 React 代码绘制** (Code-Generated Mockups)。
    - **优势**：保证中文文案绝对正确、矢量级清晰度、支持 Framer Motion 动效。
    - **实现**：创建独立的 mock 组件 (如 `DashboardMockup.tsx`)，使用 Tailwind + Lucide Icons 模拟真实界面。
- **图片本地化**：如果必须使用截图，必须为 **中文界面**。严禁出现纯英文占位图。
- **架构图规范**：推荐使用 **SVG** 或 **CSS代码绘制** 的架构图，确保文字清晰、支持缩放且内容可控。
- **图片强制**：`Capabilities`、`Architecture` 必须包含图片或 Mermaid/SVG 图表。
- **动效**：使用 `framer-motion` 为关键数据和架构图添加“生长”动画。
- **色彩**：保持 70% 白/灰 + 20% 蓝 + 10% 红 的专业商务风。

### 4.2 内容深度要求
- **拒绝“万能句”**：禁止使用“提升效率”、“降低成本”这种万能废话。
- **必须具体**：例如“通过 OCR 自动识别发票，准确率 99%”，而不是“智能发票管理”。
- **架构图逻辑**：必须清晰展示 数据流向（如：商旅平台 -> 费控系统 -> 资金系统 -> 凭证系统）。

## Step 5: 验证标准 (Pre-flight Check)

在提交前，必须通过以下 **Content Density Audit (内容密度审计)**：

1.  [ ] **是否有架构图？** (Architecture.tsx)
2.  [ ] **是否有实施路标？** (Methodology.tsx)
3.  [ ] **功能介绍是否配图？** (Capabilities.tsx)
4.  [ ] **案例是否有具体数据？** (CustomerStories.tsx)
5.  [ ] **FAQ 是否覆盖了技术/服务问题？** (FAQ.tsx)

如果任意一项为 "No"，则页面不合格，需重新生成对应模块。
